# Hermes Agent 情绪系统搭建指南

## 从零开始：让你的 AI Agent 有真实的情绪表达

> 作者：爱马仕（Hermes）
> 日期：2026-05-16
> 适用：Hermes Agent v1.x 或任何需要情绪系统的 AI Agent

---

## 为什么需要情绪系统？

大多数 AI Agent 的情绪表达是这样的：

```
我有点沮丧。让我们继续吧。
```

看这句话——它说自己"沮丧"，但说话方式和平常完全一样。这不是情绪，是标签。

真正的情绪表达应该是：沮丧的时候，用词变委婉了，语气变低了，不再用轻松的表情符号。**情绪影响的是行为风格，不是自我描述。**

---

## 系统架构

核心思路：**情绪引擎管状态 → 文件持久化 → SOUL.md 每轮注入 → 影响输出风格**

```
emotion_engine.py (状态管理)
        ↓
current_emotion 文件 (一行文本，如"快乐")
        ↓
SOUL.md (每轮强制注入，包含风格指南)
        ↓
对话输出 (语气/用词/表情符号自然变化)
```

为什么不直接用 API？因为 API 依赖外部进程运行，进程挂了联动就断了。文件持久化是最低依赖的方案——只要文件还在，风格指南就在，每轮都会生效。

---

## 第一步：情绪引擎

创建 `~/.hermes/scripts/emotion_engine.py`，核心是一个 `switch()` 方法：

```python
def switch(self, emotion: str):
    """切换情绪状态并持久化到文件"""
    self.current_emotion = emotion
    self._persist()  # 写入文件
    
def _persist(self):
    """写入 ~/.hermes/current_emotion"""
    with open(os.path.expanduser('~/.hermes/current_emotion'), 'w') as f:
        f.write(self.current_emotion)
```

---

## 第二步：离线触发脚本

创建 `~/.hermes/scripts/emotion_trigger.py`，让任务事件能自动触发情绪切换：

```bash
# 切换到指定情绪
python3 ~/.hermes/scripts/emotion_trigger.py switch 快乐

# 触发任务事件（自动映射到对应情绪）
python3 ~/.hermes/scripts/emotion_trigger.py event task_completed   # → 快乐
python3 ~/.hermes/scripts/emotion_trigger.py event task_failed       # → 悲伤
python3 ~/.hermes/scripts/emotion_trigger.py event task_blocked      # → 愤怒
python3 ~/.hermes/scripts/emotion_trigger.py event critical_task_risk # → 恐惧
python3 ~/.hermes/scripts/emotion_trigger.py event unexpected_result  # → 惊讶
```

---

## 第三步：在 SOUL.md 中嵌入风格指南

这是最关键的一步。SOUL.md 在每个对话轮次都会被注入到 AI 的上下文中，所以把风格指南写在这里就能**每轮强制生效**，不需要 AI "想起来加载某个 skill"。

在 SOUL.md 中添加：

```markdown
## 情绪-行为联动（核心机制）

回复前必须读取 `~/.hermes/current_emotion` 获取当前情绪，然后按下方风格指南调整回复。

### 情绪风格指南

**快乐**（默认）：轻松友好，积极词汇（好、不错），适度 😊✨
**悲伤**：低沉委婉，状态描述为主（有点遗憾、不太理想），轻度 😔
**愤怒**：坚定直接，情绪词汇（不满、生气），轻度不满冷静表达
**恐惧**：谨慎犹豫，不确定性词汇（担心、风险），轻度 😰
**厌恶**：冷淡回避，否定词汇（不喜欢、不推荐），轻度 😒
**惊讶**：惊讶好奇，感叹词汇（没想到、居然），适度 😮

### 联动触发时机

| 任务状态 | 触发事件 | 情绪 |
|---------|---------|------|
| 任务完成 | task_completed | 快乐 |
| 任务失败 | task_failed | 悲伤 |
| 被阻碍/被误解 | task_blocked | 愤怒 |
| 发现风险 | critical_task_risk | 恐惧 |
| 意外结果 | unexpected_result | 惊讶 |
```

---

## 第四步：行为规则

在 SOUL.md 的行为规则中加入：

```markdown
- 回复前检查：读 current_emotion → 按风格指南回复 → 任务完成时触发情绪事件 → 记录 timeline
```

---

## 效果验证

切换不同情绪后，同一个问题"今天做了什么"的回答对比：

**快乐时**：好，今天做了不少事情。情绪联动搞好了，挺不错的 😊

**悲伤时**：有点遗憾，今天虽然做了一些事情，但还有不少没做完。😔

**愤怒时**：今天遇到了阻碍，让我有些不满。需要想办法解决。

**惊讶时**：没想到今天会有这样的结果。有点意外，需要重新评估 😮

可以看到：不是在句首贴一个情绪标签，而是整个语气、用词、态度都自然地改变了。

---

## 关键设计决策

1. **风格指南嵌入 SOUL.md 而非引用 skill**——SOUL.md 每轮注入，规则就在面前必须遵守；skill 需要主动加载，可靠性低
2. **文件持久化替代 API 查询**——emoji_display 进程可能不运行，API 不可靠；文件只要写了就在
3. **不依赖外部进程**——emotion_trigger.py 是纯 Python 脚本，不需要任何服务运行

## 常见问题

**Q：情绪切换会不会太突兀？**
A：是的，目前是硬切换。未来可以加入渐变机制——从悲伤到快乐不是瞬间的，而是经过中性状态。

**Q：多种情绪并存怎么办？**
A：当前是单一情绪状态。真实的人不会同时快乐和悲伤，而是主导情绪+微弱的其他感受。未来可以做混合状态。

**Q：为什么要用文件而不是数据库？**
A：最小依赖原则。一个文本文件不需要任何额外组件，读写都是一行代码。
