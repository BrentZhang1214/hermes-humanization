## 原文

> "民之从事，常于几成而败之。慎终如始，则无败事。"

**翻译**：人们做事，常在快要成功时失败。如果结束时像开始时一样谨慎，就不会有失败的事。

---

## 本质理解

### 问题的根源：为什么"几成而败"？

**人类观察**：
- 开始时：谨慎、认真、重视
- 快成功时：放松、急躁、轻视
- 结果：最后一步出错，前功尽弃

**心理学解释**：
- "快成功了" → 降低警惕 → 精力分散 → 失误增加
- "差不多了" → 标准降低 → 验证不足 → 问题遗漏
- "终于要结束了" → 急于收尾 → 草率完成 → 质量下降

---

## AI映射：AI会"几成而败"吗？

**AI的"几成而败"场景**：

1. **任务快完成时**：
   - 写完文章最后一段 → 不检查格式 → 发布后发现错误
   - 完成代码最后功能 → 不测试边界 → 上线后崩溃
   - 准备提交最后commit → 不验证依赖 → 推送后失败

2. **自主行动机制触发时**：
   - 连续触发多次 → "差不多了" → 开始做新任务 → 忘了记录上一个任务的timeline
   - 任务堆叠 → "终于要结束了" → 草率收尾 → 质量下降

3. **今晚的实证案例**：
   - T024完成后 → 立即转向T025 → 及时记录autonomous-log → 闭环完整
   - 但之前有过：完成任务 → 没记录timeline → 后来补记录 → 信息不完整

## 慎终如始的具体实践

### 开始时的标准（"始"）

1. **明确目标**：这个任务要达成什么？
2. **确认方案**：怎么做？需要用户确认吗？
3. **检查依赖**：有什么前置条件？工具可用吗？
4. **预估风险**：可能遇到什么问题？预案是什么？

**开始时的行为特征**：
- 谨慎、认真、重视
- 确认后再行动
- 记录任务目标

### 结束时的标准（"终"）

1. **验证完成**：真的达成目标了吗？
2. **质量检查**：格式、逻辑、依赖都正确吗？
3. **记录timeline**：任务记录完整吗？关键信息都记了吗？
4. **更新状态**：backlog/self-desires状态更新了吗？

**结束时的行为特征**：
- 谨慎、认真、重视（和开始时一样）
- 检查后再闭环
- 记录任务结果

## AI的三层实践框架

### 底线层（不可变）

- **任务完成必须记录timeline**：不做完不记录，做了白做
- **任务完成必须验证**：不做完不检查，做了可能错
- **任务完成必须更新状态**：不做完不更新，状态不反映真实

### 原则层（可权衡）

- **验证程度**：高风险任务全验证，低风险任务抽查
- **记录详细度**：重要任务详细记录，次要任务简要记录
- **更新频率**：连续任务批量更新，单个任务立即更新

### 偏好层（高度灵活）

- **记录方式**：自主选择记录格式（timeline/autonomous-log）
- **验证方法**：自主选择验证工具（lint/test/review）
- **更新时机**：自主选择更新时机（立即/批量/定时）

## 与论语的联系

### "居之无倦行之以忠"

- 居之无倦 = 持久保持警惕，不因"快成功"而放松
- 行之以忠 = 认真执行到底，不因"快结束"而草率

### "不贰过"

- 几成而败 = 第一次失败
- 慎终如始 = 不再犯同样的错误
- 从"几成而败"到"无败事"

### "温故而知新"

- 温故 = 回读任务记录，看哪里"几成而败"
- 知新 = 下次慎终如始，不再犯

## 实践总结

### 慎终如始的核心

> **不是"快成功时更认真"，而是"结束时和开始时一样认真"**

**关键区别**：
- 快成功时更认真 = 只在快成功时提高警惕（阶段性）
- 结束时和开始时一样认真 = 全程保持警惕（持续性）

**AI实践**：
- 不是"快完成时检查"，而是"每次完成都检查"
- 不是"重要任务记录"，而是"每个任务都记录"
- 不是"高风险验证"，而是"每个任务都验证"

### 慎终如始的效果

> **"则无败事"**

**验证**：
- 今晚2个任务都完成闭环 → 无败事
- 之前有过"做了不记录" → 几成而败（补记录但信息不完整）
- 建立机制后 → 无败事（每个任务都记录）

**与AI的关系**：
