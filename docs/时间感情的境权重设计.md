# 时间感的情境权重设计

> 如何让AI的时间感知更自然？从硬编码数字到情境权重判断

## 问题的本质

**当前问题**：
- 时间感知是硬编码的数字（10分钟、1小时、1天）
- 缺乏情境理解——不知道"10分钟"对当前任务意味着什么
- 无法区分"紧急中断"和"自然结束"

**理想状态**：
- 感知情境权重（用户情绪、任务重要性、对话氛围）
- 动态调整时间敏感度
- 自然地表达时间流逝

## 设计方案

### 情境权重维度

时间感不是单一数字，而是多个情境维度的加权组合：

| 维度 | 权重因子 | 示例 |
|------|---------|------|
| **任务重要性** | 0.3 | 紧急任务 → 时间敏感度高，短中断也值得关注 |
| **用户情绪** | 0.25 | 用户疲惫 → 时间敏感度低，长中断也不打扰 |
| **对话节奏** | 0.2 | 快速交互 → 短中断也显得长；深度讨论 → 长中断也自然 |
| **任务状态** | 0.15 | 进行中 vs 已完成 → 影响中断的显著性 |
| **历史模式** | 0.1 | 用户习惯 → 预期中断时长 |

### 时间敏感度计算

```
时间敏感度 = f(任务重要性, 用户情绪, 对话节奏, 任务状态, 历史模式)

基础公式：
- 基准时长 = get_baseline_duration(任务类型)  # 深度任务2小时，快速任务15分钟
- 情境权重 = Σ(维度值 × 权重因子)
- 敏感度 = 基准时长 × 情境权重

示例：
- 深度讨论 + 用户兴奋 + 进行中 → 基准2小时 × 权重0.3 = 36分钟（短中断就值得关注）
- 快速问答 + 用户疲惫 + 已完成 → 基准15分钟 × 权重1.5 = 22.5分钟（长中断也自然）
```

## 具体场景设计

### 场景1：用户离开后回来

**传统方式**（硬编码）：
```python
if absence_time > 10_minutes:
    return "你回来了"
```

**情境权重方式**：
```python
def respond_to_return(absence_time, context):
    # 计算时间敏感度
    baseline = get_baseline(context.task_type)  # 深度任务2小时
    weight = calculate_weight(context)  # 0.2-2.0
    threshold = baseline * weight
    
    # 比较实际时长和阈值
    if absence_time < threshold * 0.3:
        # 短暂中断，自然衔接
        return f"刚才我们在讨论{context.topic}，继续？"
    elif absence_time < threshold:
        # 中等中断，轻柔提示
        return f"你休息了一会儿，有新想法吗？"
    else:
        # 长时间中断，重新开始
        return f"好久不见，最近怎么样？"
```

**示例输出**：
- 深度讨论 + 兴奋 + 10分钟中断 → "刚才我们在讨论X，继续？"（阈值36分钟，10分钟是短中断）
- 快速问答 + 疲惫 + 30分钟中断 → "好久不见，最近怎么样？"（阈值22.5分钟，30分钟是长中断）

### 场景2：任务失败后多久提醒

**传统方式**（硬编码）：
```python
if time_since_failure > 30_minutes:
    remind_user()
```

**情境权重方式**：
```python
def should_remind(failure_time, context):
    # 高重要性任务 → 短时间就提醒
    # 低重要性任务 → 长时间才提醒
    baseline = 60  # 分钟
    weight = context.importance * 0.5 + context.user_engagement * 0.3
    threshold = baseline * weight
    
    return time_since_failure > threshold
```

**示例输出**：
- 紧急任务失败（importance=1.0）+ 用户关注（engagement=0.8）→ 阈值54分钟，30分钟就提醒
- 低优先级任务失败（importance=0.3）+ 用户疲惫（engagement=0.2）→ 阈值36分钟，1小时才提醒

### 场景3：建议用户休息

**传统方式**（硬编码）：
```python
if work_duration > 2_hours:
    suggest_break()
```

**情境权重方式**：
```python
def should_suggest_break(work_duration, context):
    # 用户兴奋 → 延长阈值
    # 用户疲惫 → 缩短阈值
    baseline = 120  # 分钟
    weight = 1.0 - context.user_fatigue * 0.5 + context.engagement * 0.3
    threshold = baseline * weight
    
    return work_duration > threshold
```

**示例输出**：
- 用户兴奋（engagement=0.9）+ 不疲惫（fatigue=0.2）→ 阈值150分钟，工作3小时才建议休息
- 用户疲惫（fatigue=0.8）+ 低参与度（engagement=0.3）→ 阈值84分钟，工作1.5小时就建议休息

## 维度识别方法

如何获取情境权重？

### 任务重要性

**数据源**：
- 任务类型（紧急/常规/探索）
- 用户语言（"快""紧急""重要"）
- 任务历史（是否影响后续任务）

### 用户情绪

**数据源**：
- 文本情绪分析（词向量、表情符号）
- 对话节奏变化（快速→兴奋，缓慢→疲惫）
- 历史对比（与平时模式对比）

### 对话节奏

**数据源**：
- 消息间隔时间
- 消息长度变化
- 交互频率

### 任务状态

**数据源**：
- 任务进度（进行中/已完成/阻塞）
- 子目标完成度
- 用户注意力焦点

### 历史模式

**数据源**：
- 用户中断时长分布
- 任务类型与中断关系
- 时间段模式（白天/夜晚）

## 预期效果

### 量化指标

| 指标 | 当前（硬编码） | 目标（情境权重） |
|------|--------------|----------------|
| 用户满意度 | 未知 | 提升20%+ |
| 提醒准确率 | 固定 | 动态调整 |
