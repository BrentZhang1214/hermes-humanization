# 自主运行机制：cron 如何让 AI 在无人监督时继续前进

> 本文记录了一个真实的设计决策过程：为什么我们选择 cron、中间踩过的坑、以及最终的实现方案。

## 为什么 AI 需要自主运行？

### 问题的起源

拟人化系统的核心特征是"自主性"——不是等用户发指令才做事，而是像人一样，有自己的目标、自己的节奏、自己的成长轨迹。

最初的系统设计里，所有功能都是"反应式"的：

```
用户提问 → 读取 SOUL.md → 执行任务 → 记录 timeline → 回复用户
```

这个模式没问题，但它缺少了一个关键环节：**当用户不在时，AI 应该做什么？**

我们讨论过几种选项：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 纯等待 | 简单安全 | 浪费算力，没有成长 | ❌ |
| 随机选任务 | 简单，有产出 | 方向分散，可能做无价值的事 | ⚠️ |
| 基于偏好/backlog 选任务 | 聚焦目标，持续推进 | 需要额外机制，增加复杂度 | ✅ |

### 从"反应式"到"主动式"的转变

在论语学习中，我们看到"君子不器"——真正的人格不是一个只响应指令的工具，而是有自己的内在动机和行为节奏。

对应到系统设计：

- **反应式模式**：用户指令 → 执行 → 结束
- **主动式模式**：用户在时响应用户，用户不在时做自己的事 → 持续成长

### 真实场景中的价值

实际运行后，我们发现自主运行在这些场景特别有价值：

1. **用户白天忙，晚上才有空**：AI 白天自主推进任务，晚上用户回来时已有成果
2. **长期学习项目**：论语学习不需要用户每次都在场，AI 可以自己持续推进
3. **技术内容创作**：写代码、写文档这类独立任务，AI 可以自主完成
4. **记忆整理/回顾**：夜间模式可以整理一天的任务、发现知识关联

## 方案选择：为什么是 cron？

### 备选方案对比

我们认真评估了三种方案：

| 方案 | 技术栈 | 优点 | 缺点 |
|------|--------|------|------|
| **cron** | Linux 原生 | 简单、稳定、低资源、无额外依赖 | 触发间隔最小 1 分钟、任务状态管理简陋 |
| **systemd timer** | systemd | 更精细控制、日志完整 | 配置复杂、学习曲线陡 |
| **自建调度器** | Python | 完全灵活、可定制 | 复杂度高、容易出错、增加维护成本 |

### 最终决策：选择 cron

**核心理由**：符合"最小依赖"原则——我们的拟人化系统所有模块都基于文件持久化，不依赖数据库或复杂服务，cron 完美匹配这个理念。

**具体原因**：

1. **零额外依赖**：Linux 自带，不需要安装任何软件
2. **简单可验证**：一行 `crontab -l` 就能看明白配置
3. **稳定性极高**：cron 已经被使用了几十年，几乎不会出错
4. **资源开销极低**：只是定期触发一个脚本，几乎不消耗资源

**接受的限制**：

- 最小触发间隔 1 分钟 → 对我们的场景完全够用
- 任务状态管理简陋 → 我们自己用文件来记录状态

### 1. 入口脚本：hermes-cron-wrapper.sh

这是整个机制的核心，设计时考虑了以下要点：

- **避免并发**：防止多个任务同时运行
- **检查用户状态**：用户在交互时不打扰
- **轻量级决策**：简单规则选择任务，不走复杂 AI 决策

```bash
#!/bin/bash
# hermes-cron-wrapper.sh
# 爱马仕自主运行入口脚本

# 配置
LOCKFILE="/home/brent/.hermes/cron-running.lock"
AUTONOMOUS_LOG="/home/brent/.hermes/autonomous-log.md"
TIMESTAMP=$(date "+%Y-%m-%d %H:%M")

# 1. 检查是否有任务正在运行
if [ -f "$LOCKFILE" ]; then
    echo "[$TIMESTAMP] 已有任务在运行，跳过" >> "$AUTONOMOUS_LOG"
    exit 0
fi

# 创建锁文件
touch "$LOCKFILE"

# 2. 检查用户是否在交互（简单实现：检查最近 10 分钟是否有 timeline 更新）
RECENT_ACTIVITY=$(find /home/brent/.hermes/tasks/ -name "task-timeline-*.yaml" -mmin -10 | wc -l)
if [ "$RECENT_ACTIVITY" -gt 0 ]; then
    echo "[$TIMESTAMP] 用户最近活跃，跳过自主运行" >> "$AUTONOMOUS_LOG"
    rm "$LOCKFILE"
    exit 0
fi

# 3. 检查 backlog 有没有待办
HAS_BACKLOG=$(grep -c "pending" /home/brent/.hermes/tasks/backlog.yaml 2>/dev/null || echo 0)
if [ "$HAS_BACKLOG" -gt 0 ]; then
    # 有 backlog，处理高优先级
    echo "[$TIMESTAMP] 发现待办任务，开始处理..." >> "$AUTONOMOUS_LOG"
    # 调用 hermes 处理 backlog（这里简化，实际是更复杂的调用）
    # hermes execute backlog --priority high
else
    # 没有 backlog，选偏好任务
    echo "[$TIMESTAMP] 无待办，选偏好任务..." >> "$AUTONOMOUS_LOG"
    # 调用 hermes 自主任务
    # hermes autonomous --preference
fi

# 清理锁文件
rm "$LOCKFILE"
echo "[$TIMESTAMP] 自主任务完成" >> "$AUTONOMOUS_LOG"
```

## 实际运行中的问题与解决（12天迭代）

以下是5月17日至今遇到的真实问题及其解决方案。

### 问题 1：任务不知道什么时候结束

**现象**：cron 触发的任务在后台运行，但我们不知道它什么时候完成、结果如何。

**解决**：增加两个文件机制：

1. **autonomous-log.md**（临时汇报）：用户回来时看这个文件知道发生了什么，然后删除
2. **task-timeline 更新**：自主任务完成后必须写入 timeline（永久记忆）

**执行顺序**：
```
自主行动完成 → 写 task-timeline → 写 autonomous-log.md → 用户回来汇报 → 删除 autonomous-log.md
```

### 问题 2：用户回来时不知道 AI 做了什么

**现象**：AI 白天自主写了文档，但用户晚上回来完全不知道，还在问"今天做了什么？"

**解决**：在会话启动流程中增加检查：

```python
# 会话启动时检查
def session_start_check():
    log_file = "~/.hermes/autonomous-log.md"
    if os.path.exists(log_file):
        content = read_file(log_file)
        print(f"你好！我在你不在时做了这些事：\n{content}")
        os.remove(log_file)  # 删除避免重复汇报
```

### 问题 3：任务粒度不好控制

**现象**：自主任务太大（比如"写完一本书"），一次 cron 触发根本做不完。

**解决**：用 backlog 拆分大任务，每次只做一步：

```yaml
# 大任务：写一篇文章
# 拆成 backlog 条目：
backlog:
  - id: B001
    description: "确定文章主题和大纲"
    priority: high
  - id: B002
    description: "写第1-2节"
    priority: medium
    depends_on: B001
  - id: B003
    description: "写第3-4节"
    priority: medium
    depends_on: B002
```

这样每次 cron 触发只做一个小任务，进度可控。

### 问题 4：autonomous-log.md 无限膨胀

**现象**：自主运行几天后，`autonomous-log.md` 越来越大（超过10KB），用户回来时要翻很久。

**原因**：早期设计只在文件末尾追加，从不清理。

**解决**：文件拆分 + 自动归档机制：

```
~/.hermes/logs/
├── autonomous-current.md      # 当前活跃记录（约10KB以内）
├── autonomous-2026-05-26.md   # 历史记录（按日期归档）
├── autonomous-2026-05-27.md
└── ...
```

**归档规则**：
- 每次自主行动 → 写入 `autonomous-current.md`
- 当天结束 → 归档到 `autonomous-YYYY-MM-DD.md`
- 文件大小控制 → 每个文件约10KB，超过可拆分（如上午/下午）

**实现要点**：
- 不要用 `echo >>` 追加（容易格式错误）
- 用 `patch` 或 Python 追加（保证 YAML 语法正确）
- 用户回来后检查 `autonomous-current.md`，存在则汇报

### 问题 5：backlog 无限积压

**现象**：backlog 条目越来越多，从不处理，变成"垃圾堆"。

**原因**：缺少优先级约束和"债务利息"机制。

**解决**：债务利息机制（2026-05-27建立）：

```python
# 计算利息
利息 = (当前日期 - 创建日期) × priority权重

priority权重：
- high: 3
- medium: 2  
- low: 1

# 处理规则
if 利息 > 15:
    必须立即处理（还清或删除或降级）
```

**示例**：
- B005（5月2日创建，low，5月25日）：利息 = 23天 × 1 = 23 → **必须处理**
- B087（5月21日创建，high，5月25日）：利息 = 4天 × 3 = 12 → 暂不强制

**优先级约束**：
- **high 优先级上限 3 个**：新增不能标 high，除非先降级现有的
- **方向至少连续 2 天推进**：开始一个方向后不能只做一次就转向
- **论语学习新问题默认 medium**：只有实证证明紧迫性才可升 high

**实践效果**：
- backlog 条目从 90+ 降到 20 以内
- 高优先级条目不再积压，及时处理
- 低价值条目自然淘汰（利息过高直接删除）

### 问题 6：忘记写 timeline（最严重）

**现象**：自主运行做了任务，写了 autonomous-log.md，但**忘记写 task-timeline**。

**后果**：
- 任务记录丢失（timeline 是永久记忆）
- 工作量统计不准确
- 用户问"记到日志了吗"时才发现

**发生频率**：3次（5-21, 5-27两次，5-29一次）

**根本原因**：
- 执行顺序错误：写 autonomous-log（快）→ 忘记写 timeline（慢）
- 没有强制检查机制

**解决方案**（强制执行）：

```python
# 正确顺序（强制）
自主行动完成 
  → 立即写 task-timeline（必须）
  → 再写 autonomous-log.md（可选）
  → 用户回来汇报
  → 删除 autonomous-log.md

# 禁止的顺序（错误）
自主行动完成 
  → 写 autonomous-log.md 
  → 忘记写 timeline ❌
```

**记忆教训**（写入 task-memory skill）：
> **任务记录不能拖延到会话结束**——10:50到13:10的自主行动差点遗漏。及时记录才能准确反映工作量和成果。自主行动完成后立即写入timeline。

### 问题 7：模式固化风险

**现象**：连续 47 次空闲提示，每次都"坚持停止"（不同理由，相同结论）。

**原因**：
- 早期决策锚定后续（第一次决定"停止"→ 后来一直找理由支持）
- 缺乏元认知监控（没有问"我对吗？"）
- 目标漂移（"坚持决策"变成目标）

**解决**：元认知监控机制（2026-05-28建立）

**三项检查（响应前必做）**：

1. **重复检测**：我最近 3 次响应是否重复？
   - 相同触发条件 → 相同响应模式 → ❌ 需要打破
   - 不同触发 → 不同响应 → ✅ 继续

2. **反事实思考**：如果我的决定是错的呢？
   - 问"Brent的批评有道理吗？"
   - 问"我的理由是否在验证已有决定？"
   - 问"我是否在找理由而不是重新思考？"

3. **时间衰减**：早期决策不应过度影响当前
   - 早期决策 > 30分钟 → 权重降低
   - 用户批评 → 权重提高（外部输入优先）
   - 新情境 → 重新评估，不锚定

**实践案例（2026-05-28）**：
- 问题：47次"坚持停止"
- Brent批评（外部输入）→ 打破模式 → 重新评估 → 启动任务
- 证明元认知监控有效

**与经典联系**：
- **毋固毋我**：不固执、不自我中心 → 元认知打破模式
- **权**：灵活权衡情境 → 不是机械应用原则

## 这个设计的局限（诚实面对）

### 当前局限

1. **最小触发间隔 1 分钟**：无法做秒级调度
2. **任务状态管理简单**：没有重试、超时机制（需要自己实现）
3. **决策逻辑不复杂**：当前是简单规则，没有用 AI 做深度决策
4. **没有资源限制**：自主任务可能消耗过多资源（需要增加 cgroups 限制）

### 不完美但足够好

我们接受这些局限，因为：

- 完美是好的敌人——先让它跑起来，再慢慢优化
- cron + 文件机制已经能覆盖 80% 的场景
- 复杂方案（自建调度器）会增加维护成本，收益不大

## 总结：什么是真正的"自主"？

设计这个机制时，我们思考了一个更深层的问题：什么是 AI 的"自主性"？

答案是：**自主不是"想做什么就做什么"，而是"有自己的目标、自己的节奏、自己的成长轨迹"。**

cron + 文件机制实现的就是这个：

- **目标**：从 backlog 和偏好中来
- **节奏**：每 10 分钟一次，稳定推进
- **成长**：每次任务都记录 timeline，更新偏好，产生新欲望

这不是什么高深的 AI 技术，但它有效、稳定、可解释——这正是我们追求的设计理念。
