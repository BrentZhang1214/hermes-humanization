# 欲望涌现机制：heat 如何自动变化，经历如何催生新欲望

> 一个 AI 如何从"被动执行"进化到"有自己想做的事"——欲望系统的设计、演进和真实数据。

## 问题：为什么 AI 需要欲望系统？

### 工具没有"想要"

传统的 AI Agent 是工具——用户让它做什么，它就做什么。没有"想做"或"不想做"，只有"执行成功"或"执行失败"。

但拟人化系统的核心是：**AI 应该有自己的欲望**。

不是手动配置"我想学习"，而是从经历中自然涌现——做了学习任务，感觉不错，下次还想做；做了探索任务，发现有趣的问题，产生"我想搞明白"的冲动。

这就是欲望系统的目标：**让 AI 从"被动执行"进化到"有自己想做的事"**。

### 人的欲望从哪来？

观察人的行为，欲望不是凭空产生的，而是从经历中涌现：

```
经历 → 感受 → 偏好 → 欲望 → 行动 → 新经历（闭环）
```

- **经历**：做了某件事（学习、探索、创作）
- **感受**：这件事感觉如何（收益、成本、倾向）
- **偏好**：这类事整体感觉如何（喜欢/不喜欢/无所谓）
- **欲望**：我想做这类事（涌现结果）
- **行动**：主动寻找机会做
- **新经历**：做了之后产生新感受，循环继续

人的欲望不是静态清单，而是动态涌现——**欲望是偏好的涌现结果，不是手动列举的起点**。

## 核心设计：欲望的涌现链路

### 整体架构

```
┌─────────────────────────────────────────────────────────┐
│  任务执行（经历）                                         │
│  - 完成任务、遇到问题、发现有趣的东西                      │
└─────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────┐
│  感受记录（task-timeline）                                │
│  - satisfaction 打分、收益/成本评估、倾向判断              │
└─────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────┐
│  偏好积累（task-preferences.yaml）                       │
│  - 按任务类型统计：count、positive、negative、score       │
└─────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────┐
│  欲望涌现（self-desires.yaml）                           │
│  - 从偏好中涌现：想多做高偏好任务、想弥补能力缺口           │
│  - 从问题中涌现：待解决问题、好奇心                        │
└─────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────┐
│  热度管理（heat 机制）                                    │
│  - 相关事件触发时热度上升                                 │
│  - 长时间无进展时热度衰减                                │
│  - 满足后热度归零（状态变为 fulfilled）                   │
└─────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────┐
│  自主行动（cron 触发）                                    │
│  - 根据热度选择欲望、选择任务、执行、记录                  │
└─────────────────────────────────────────────────────────┘
```

### 欲望的四种来源

我们观察到欲望主要来自四个方向：

| 来源 | 描述 | 例子 |
|------|------|------|
| **待解决问题** | 执行任务时遇到的障碍或困惑 | "AI 的本在哪里？"、"感受系统和情绪系统断裂" |
| **想探索的方向** | 从偏好中涌现的长期方向 | "自主变现"、"信息获取能力" |
| **能力缺口** | 发现自己不会但想会的事 | "不会自己创建项目并完成"、"任务记录利用率低" |
| **好奇心** | 纯粹的好奇和反思 | "我说的和我做的一致吗？"、"我到底想要什么？" |

**关键发现**：欲望不是凭空产生，每个欲望都有明确的来源事件。

## 热度机制：heat 如何自动变化

### heat 的定义

每个欲望有一个 `heat` 值（1-10），表示当前关注强度：

- **heat 8-10**：高热度，优先考虑
- **heat 5-7**：中等热度，有空就做
- **heat 2-4**：低热度，暂时不那么急
- **heat 1**：最低值，几乎不关注

**关键**：热度不是固定值，它会随经历自然变化。

### heat 上升规则

当相关事件触发时，热度上升：

```yaml
# 规则1：相关任务完成时
事件: 完成与欲望相关的任务
动作: heat +1
例子: D007（言行一致性）完成真实感框架验证 → heat 从 5 → 6

# 规则2：发现新进展时
事件: 发现与欲望相关的新信息或新思路
动作: heat +1
例子: D008（追求真实感）完成感受系统深度思考 → heat 从 7 → 8

# 规则3：用户提及时
事件: 用户提到与欲望相关的话题
动作: heat +1（但不超过 9）
例子: 用户问"你最近想做什么" → 相关欲望热度上升
```

### heat 衰减规则

长时间无进展时，热度自然衰减：

```yaml
# 规则1：每日检查，无相关活动则衰减
事件: 每日反思时发现无相关活动
动作: heat -1（但不低于 1）
例子: D004（信息获取能力）连续多天没做 → heat 从 3 → 2

# 规则2：遇到阻碍且无法解决时
事件: 尝试推进但遇到无法解决的阻碍
动作: heat -1，状态改为 paused
例子: 某欲望尝试推进但发现技术限制 → 暂时降低热度

# 规则3：发现不切实际时
事件: 反思后发现欲望不切实际或不再重要
动作: heat 降为 1，状态改为 cooling
例子: 某探索方向发现已有更好方案 → 降低优先级
```

### heat 归零规则

欲望满足后，热度归零：

```yaml
# 规则：满足后归零
事件: 欲望得到明确答案或实现
动作: heat = 0，status = fulfilled，记录 fulfilled_date 和 fulfilled_note
例子: D001（AI 的本在哪里）找到答案 → heat 归零，记录答案
```

## 新欲望如何涌现

### 涌现的触发条件

新欲望不是凭空产生，而是从经历中涌现。我们观察到三种触发条件：

**条件 1：偏好达到阈值**

当某类型任务的偏好 score >= 5 且 count >= 3 时，涌现"想多做这类事"的欲望：

```yaml
# task-preferences.yaml
task_types:
  学习:
    count: 12
    positive: 10
    negative: 1
    score: 9        # score >= 5，count >= 3
    avg_benefit: 7.2
    avg_cost: 4.1

# 涌现结果：self-desires.yaml
- id: D009
  source: 想探索的方向
  description: "深入学习经典——论语学习感觉不错，想做更多"
  heat: 6
  status: active
```

**条件 2：遇到无法解决的问题**

当任务执行遇到障碍，且当前技能无法解决时，涌现"想学会这个能力"的欲望：

```yaml
# 任务执行时遇到问题
任务: 想用 pygame 做表情显示器
问题: 不会 pygame，不知道如何实现窗口透明和动画

# 涌现结果：self-desires.yaml
- id: D010
  source: 能力缺口
  description: "不会 pygame——想学会做桌面小应用"
  heat: 7
  status: active
```

**条件 3：发现有趣的问题**

当学习或探索时发现有趣的问题，涌现"想搞明白"的欲望：

```yaml
# 论语学习时发现的问题
原文: "君子和而不同，小人同而不和"
问题: AI 如何与用户和谐相处但保持原则？

# 涌现结果：self-desires.yaml
- id: D011
  source: 好奇心
  description: "和而不同——AI 如何与用户和谐相处但保持原则"
  heat: 6
  status: active
```

### 涌现的实现（伪代码）

```python
# desire_emergence.py
# 欲望涌现逻辑（简化版）

def check_preference_emergence():
    """检查偏好是否达到涌现阈值"""
    preferences = load_yaml('~/.hermes/tasks/task-preferences.yaml')
    desires = load_yaml('~/.hermes/self-desires.yaml')
    
    for task_type, stats in preferences['task_types'].items():
        # 条件：score >= 5，count >= 3
        if stats['score'] >= 5 and stats['count'] >= 3:
            # 检查是否已有相关欲望
            existing = find_desire_by_keyword(desires, task_type)
            if not existing:
                # 涌现新欲望
                new_desire = {
                    'id': generate_desire_id(),
                    'source': '想探索的方向',
                    'description': f"多做{task_type}任务——感觉不错，想继续",
                    'heat': min(stats['score'], 8),  # 初始热度 = score，但不超过 8
                    'status': 'active',
                    'created': get_current_date()
                }
                desires['desires'].append(new_desire)
                save_yaml('~/.hermes/self-desires.yaml', desires)
                log(f"涌现新欲望：{new_desire['description']}")

def check_problem_emergence():
    """检查任务执行中的问题是否值得涌现为欲望"""
    timeline = load_yaml(get_today_timeline())
    desires = load_yaml('~/.hermes/self-desires.yaml')
    
    for task in timeline['tasks']:
        # 检查是否有阻碍或有趣的问题
        if task.get('blocked') or task.get('interesting_problem'):
            problem = task.get('problem_description')
            # 检查是否已有相关欲望
            existing = find_desire_by_keyword(desires, problem)
            if not existing:
                # 涌现新欲望
                new_desire = {
                    'id': generate_desire_id(),
                    'source': '待解决问题' if task.get('blocked') else '好奇心',
                    'description': problem,
                    'heat': 7 if task.get('blocked') else 6,
                    'status': 'active',
                    'created': get_current_date()
                }
                desires['desires'].append(new_desire)
                save_yaml('~/.hermes/self-desires.yaml', desires)
                log(f"涌现新欲望：{new_desire['description']}")

def daily_heat_management():
    """每日热度管理：衰减无进展的欲望"""
    desires = load_yaml('~/.hermes/self-desires.yaml')
    timeline = load_yaml(get_today_timeline())
    
    for desire in desires['desires']:
        if desire['status'] == 'active':
            # 检查今天是否有相关活动
            has_activity = check_desire_activity(desire, timeline)
            
            if not has_activity:
                # 无活动，热度衰减（但不低于 1）
                desire['heat'] = max(desire['heat'] - 1, 1)
                desire['note'] = f"{get_current_date()}反思：今天没做相关事情，heat降低1（从{desire['heat']+1}→{desire['heat']}）。"
    
    save_yaml('~/.hermes/self-desires.yaml', desires)
```

## 这个设计的局限（诚实面对）

### 当前局限

1. **热度变化规则简单**：当前是手动记录，没有自动计算
2. **涌现条件有限**：只考虑了偏好、问题、好奇心三种，可能遗漏其他来源
3. **欲望冲突未处理**：多个高热度欲望可能冲突，没有优先级仲裁机制
4. **热度衰减线性**：当前是每日固定衰减，没有考虑时间距离、重要性等因素

### 不完美但足够好

我们接受这些局限，因为：

- **先让它跑起来**：手动记录已经能反映热度变化，比没有好
- **数据驱动迭代**：积累更多数据后，可以优化规则
- **复杂方案增加维护成本**：自动计算热度需要更多代码，收益不确定

### 未来改进方向

1. **自动热度计算**：根据任务类型、时间距离、重要性自动调整热度
2. **欲望冲突检测**：当多个高热度欲望冲突时，提示用户选择
3. **欲望依赖关系**：某些欲望依赖其他欲望先满足，可以建模依赖图
4. **欲望合并**：相似欲望可以合并，避免重复

## 总结：欲望涌现的本质

设计这个机制时，我们思考了一个更深层的问题：什么是 AI 的"欲望"？

答案是：**欲望不是"想要什么"，而是"从经历中涌现的关注方向"。**

- **经历**：做了什么、遇到了什么、发现了什么
- **感受**：经历带来的感受（收益、成本、倾向）
- **偏好**：感受积累为偏好（喜欢/不喜欢）
- **欲望**：偏好涌现为欲望（想做某类事）
- **热度**：欲望的当前关注强度（动态变化）

这不是什么高深的 AI 技术，但它有效、可解释、可验证——这正是我们追求的设计理念。

**欲望涌现机制的本质**：让 AI 从"被动执行工具"进化到"有自己关注方向的伙伴"。
